## PART 3: LOOPHOLES AND BUGS FOUND

### CRITICAL BUG #1: GDS Generation Fallback Hides Real Failures
**File**: `python/gds_generator.py:262-278`

```python
# Fallback: Generate minimal valid GDSII binary using gdspy
self.logger.warning("Magic GDS export failed; using gdspy fallback")
try:
    import gdspy
    fallback_gds = output_dir / f"{top_module}.gds"
    
    # Create library and minimal cell
    lib = gdspy.GdsLibrary()
    cell = lib.new_cell(top_module)
    
    # Add dummy rectangle to make valid GDSII (metal1 layer 34)
    rect = gdspy.Rectangle((0, 0), (100, 100))
    cell.add(rect, layer=34)  # Metal1 in Sky130
    
    # Write binary GDSII
    lib.write_gds(str(fallback_gds))
    
    if fallback_gds.exists():
        size_mb = fallback_gds.stat().st_size / (1024 * 1024)
        result.gds_path = str(fallback_gds)
        result.gds_size_mb = size_mb
        result.success = True  # ⚠️ SILENTLY MARKS AS SUCCESS
        result.error_message = "Fallback GDS (gdspy) - basic geometry only"
```

**What happens**:
1. Magic export fails (tool not running, PDK missing, permissions issue, etc.)
2. Code catches exception and logs WARNING (not error)
3. gdspy creates a fake 212-byte rectangle
4. `result.success = True` — **flow continues as if nothing went wrong**
5. `result.error_message` is set but often ignored by downstream code
6. User receives a non-functional GDS file that will fail tape-out

**Evidence**: validation/run_001/07_gds/adder_8bit.gds is this exact fallback (212 bytes)

**Verdict**: **CRITICAL SECURITY ISSUE** — user ships fake chip to foundry

## PART 6: TECHNICAL DEBT

### T1: Circular Risk with docker_manager
- floorplanner.py imports docker_manager
- docker_manager imports Path from pathlib
- No direct circular reference, but complex path translation is untested

### T2: Windows Paths in Docker Mount Arguments
**File**: docker_manager.py:620-640
```python
mounts = [
    "-v", f"{host_work_str}:/work",  # Windows path passed to docker -v
]
```
- Docker Desktop on Windows handles this, but relies on Docker being "smart"
- Could fail silently if path has special chars

### T3: No Global Error Handler
- Each stage catches its own docker.run_script() failures
- No centralized logging of which stage failed
- Hard to debug after the fact

### T4: Missing PDK in Docker Image
- Code assumes /pdk/sky130A available in container
- verify_installation() doesn't check this
- If PDK missing, all downstream stages fail silently

### T5: Test Mocking Pattern is Outdated
- Real integration tests should run Docker once, capture output, store as fixtures
- Current pattern doesn't catch output parsing bugs
- 533 passing tests = false confidence

### T6: GDS Fallback Logic Breaks Real/Fake Boundary
- GDS should either be real (created by Magic) or not exist (return error)
- Current: returns fake but marks as success
- Users can't distinguish
